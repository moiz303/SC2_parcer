# Документация: `import_to_blender.py`

## Назначение
Центральный 3D-движок проекта. Отвечает за перевод структурированных JSON-данных (результатов парсинга реплея и анализа битвы) в полноценную 3D-сцену Blender, настройку анимаций, материалов, камеры и финальный рендер видео.

## Описание
Это самый объемный и сложный скрипт в проекте. Он выполняет роль «моста» между абстрактными данными и визуальной составляющей. Скрипт инкапсулирует в себе логику предобработки (поиск рабочих, расчет времени постройки), умный импорт ассетов, математический расчет траекторий и скоростей, а также управление нелинейной анимацией (NLA). 

**Особенность архитектуры:** Скрипт спроектирован для работы в двух режимах:
1. **Продакшн:** Запускается `controller_backend.py` в headless-режиме Blender через `subprocess` с передачей путей через аргументы командной строки (`sys.argv`).
2. **Ручной отладочный:** Может быть запущен напрямую внутри графического интерфейса Blender (или через консоль) для тестов. В этом случае используются «моковые» глобальные переменные в начале файла, имитирующие входные данные.

## ️ Структура (High-Level Overview)

### 1. Конфигурация и Моковые данные (Раздел 1)
*   **Глобальные переменные (`JSON_UNIFIED_PATH`, `ANALYSIS` и др.)**: Жестко заданные пути и словари. Используются **только** при ручном запуске внутри Blender. При запуске из бэкенда они перезаписываются функцией `load_config_from_args`.
*   **Константы и маппинги**: `SC2_FPS` (22.4), `BLENDER_FPS` (30), `RACE_FOLDER_MAP` (привязка рас к именам папок), словари ключевых слов (`UNIT_STATE_KEYWORDS`, `BUILDING_STATE_KEYWORDS`) для классификации анимаций.

### 2. Технические хелперы и утилиты (Раздел 3)
*   **`is_action_empty(action)`**: **Критическая функция.** Проверяет, содержит ли анимация реальное движение. *Использует новый API Blender 4.4+ (`action.layers`), с фоллбэком на старый (`action.fcurves`).*
*   **`clear_scene()`**: Агрессивная очистка сцены, кэшей и orphan-данных. Предотвращает утечки памяти при повторных запусках.
*   **`get_race_material()`**: Процедурная генерация материалов. Ищет текстуры (`.dds`, `.png`) на диске. Если не находит — создает цветной Principled BSDF на основе расы юнита.
*   **`add_nla_strip_safe()`**: Безопасное добавление полос в NLA-трек. Умеет обрезать пересекающиеся анимации и настраивать зацикливание (`force_loop`).
*   **`set_visibility_keys()`**: Анимация появления и исчезновения объектов через ключевые кадры `hide_viewport` и `hide_render`.

### 3. Настройка окружения (Раздел 4)
*   **`setup_scene_lighting()`**: Создает мир (World) и солнце (Sun) с теплым светом.
*   **`setup_ground_plane()`**: Генерирует плоскость земли с UV-разверткой, повторяющей текстуру 10x10 раз.
*   **`setup_analysis_camera()`**: Создает камеру, привязанную к пустому объекту (Pivot). Pivot вращается вокруг центра главной битвы, создавая эффект кинематографического облета.

### 4. Фазы обработки данных (Разделы 5-8)
*   **Фаза 1: Предобработка (`preprocess_data`)**. Математическая логика *до* создания объектов в Blender. Ищет рабочих (SCV/Drone/Probe), подходящих к месту стройки, «ведет» их к зданию и высчитывает точное время начала и конца строительства.
*   **Фаза 2: Загрузка ресурсов (`load_resources_from_folder`)**. Сканирует папки, импортирует нужные `.blend` файлы, фильтрует пустые анимации, классифицирует их по состояниям (birth, walk, attack и т.д.) и складывает в глобальные кэши (`MODEL_CACHE`, `ANIM_CACHE`).
*   **Фаза 3: Импорт юнитов (`import_units`)**. Дублирует модели из кэша. Для каждого юнита рассчитывает траекторию, подбирает анимацию (ходьба/атака/работа), **динамически масштабирует скорость анимации** под реальную скорость движения в игре и рассчитывает поворот корпуса (target rotation).
*   **Фаза 4: Импорт зданий (`import_buildings`)**. Обрабатывает жизненный цикл зданий, включая трансформации (например, Command Center -> Orbital Command), разбивая их на временные сегменты.

### 5. Точки входа и Рендер (Раздел 9)
*   **`import_all()`**: Главный оркестратор. Вызывает все фазы по порядку, экспортирует сцену в `.obj` для превью (сигнал `__GLB__`), настраивает рендер и запускает `bpy.ops.render.render(animation=True)`.
*   **`load_config_from_args()`**: Парсит `sys.argv` после разделителя `--`, чтобы получить JSON-конфиг от `controller_backend.py`.

## ️Важные технические детали (Для разработчиков)

### 1. Жесткое требование Blender 4.4+ (Критично для UI)
В функции `is_action_empty` используется проверка `hasattr(action, 'layers')`. Это **новый API Layered Actions**, появившийся в Blender 4.4. 
*   **Проблема:** Если пользователь запустит приложение с Blender 4.3 или ниже, скрипт не упадет сразу (сработает фоллбэк на `fcurves`), но обработка анимаций может работать некорректно или очень медленно на сложных ассетах.
*   **Рекомендация:** В `frontend_main.py` (или в `controller_backend.py` при вызове `find_blender_executable`) **необходимо добавить проверку версии Blender**. Если версия < 4.4, нужно либо блокировать запуск с понятным сообщением для пользователя, либо показывать предупреждение.

### 2. Умный расчет скорости анимации (Animation Scaling)
В `import_units` реализована отличная фича: скрипт не просто проигрывает анимацию ходьбы. Он вычисляет *фактическую* скорость юнита на отрезке пути (`game_dist / game_time_seconds`) и сравнивает её с базовой скоростью из `speeds.json`. 
Если юнит в игре бежал быстрее, чем в стандартной анимации Blender, скрипт увеличивает `strip.scale` (и наоборот). Это делает движение армии визуально достоверным.

### 3. Математика поворота (Shortest Path Rotation)
Чтобы юниты не крутились на 350 градусов, когда им нужно повернуться на 10, в `import_units` используется математика нормализации угла:
```python
diff = target_rot - prev_rot
if diff > math.pi: diff -= 2 * math.pi
elif diff < -math.pi: diff += 2 * math.pi
```
Это гарантирует, что модели всегда поворачиваются по кратчайшей дуге.

### 4. Логика «Строительства» (Worker-Building Matching)
В `preprocess_data` реализован жадный алгоритм привязки рабочих к зданиям. Он ищет рабочего той же расы, который подошел ближе всего (радиус < 15.0), «обрезает» его траекторию ровно в точке стройки и добавляет синтетические ключевые кадры. 
*Нюанс для Зергов:* Если здание строит Drone, скрипт искусственно убивает его (`matched_builder["died_frame"] = construction_start`), так как в SC2 дрейны «вливаются» в здание и исчезают.

### 5. Двойной режим инициализации (Mock vs CLI)
*   Если вы запускаете скрипт через `Blender -> Text Editor -> Run Script`, он использует пути из начала файла (Раздел 1). Это идеально для отладки визуала без запуска всего Python-бэкенда.
*   Если скрипт вызывается через `blender --background --python import_to_blender.py -- payload.json output.mp4`, функция `load_config_from_args` найдет `payload.json`, прочитает его и **перезапишет** все глобальные переменные. Моковые данные в этом случае игнорируются.

### 6. Обработка трансформаций зданий
Здания в SC2 часто эволюционируют (Barracks -> Barracks Reactor / Flying). Скрипт в `import_buildings` разбивает жизнь такого здания на **сегменты**. Для каждого сегмента создается отдельный объект в Blender, настраивается анимация морфинга (переход из одной модели в другую), а видимость предыдущего сегмента скрывается ровно в момент трансформации.

### 7. Протокол связи с Frontend
Скрипт активно использует `ui_progress("import_scene", percent, message)`. 
*   На 85% он делает `bpy.ops.wm.obj_export` и выводит в stdout строку `__GLB__`. Это сигнал для `controller_backend`, который передает его на фронтенд, чтобы показать пользователю 3D-превью сцены до начала долгого рендера.
*   Сам рендер (Eevee, H.264, 1280x720) запускается в самом конце. Прогресс рендера перехватывается не этим скриптом, а `_RenderProgressTracker` в `controller_backend` по строкам `Video append frame`.

### 8. Очистка сцены и утечки памяти
Функция `clear_scene()` написана с учетом «болей» Blender Python API. Она не просто удаляет объекты, но и чистит `bpy.data.actions`, `bpy.data.libraries`, и принудительно вызывает `bpy.ops.outliner.orphans_purge`. Это критически важно, так как при импорте сотен `.blend` файлов Blender склонен оставлять «осиротевшие» меши и материалы в памяти, что при пакетной обработке привело бы к крашу или диким тормозам.
