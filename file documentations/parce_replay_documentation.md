# Документация: `parce_replay.py`

## Назначение
Модуль отвечает за низкоуровневый парсинг реплеев StarCraft II (с использованием библиотеки `sc2reader`) и преобразование сырых игровых событий в структурированный, унифицированный JSON-формат. 

## Описание
Скрипт извлекает из реплея метаданные, а также траектории и жизненные циклы всех зданий и юнитов. Он объединяет разрозненные события (рождение, перемещение, апгрейд/трансформация, смерть) в единую логическую сущность. Поддерживает фильтрацию "шума" при движении юнитов, нормализацию координат относительно центра карты и интеграцию с UI для отслеживания прогресса.

---

## Структура (High-Level Overview)

### 1. Модели данных (Dataclasses)
Используются для хранения и последующей сериализации в JSON (`to_dict()`).
*   **`Position`**: Точка на карте (frame, time, x, y, z).
*   **`BuildingTransform`**: Факт эволюции здания (например, Lair -> Hive).
*   **`BaseEntity`**: Базовый класс для юнитов и зданий (ID, тип, владелец, раса, время появления/смерти).
*   **`Building`**: Наследуется от `BaseEntity`. Содержит список `frames` (позиции) и `transforms` (апгрейды).
*   **`Unit`**: Наследуется от `BaseEntity`. Содержит список `positions` и флаг `is_army`.

### 2. Основной класс: `UnifiedReplayParser`
Движок парсинга. Инкапсулирует всю логику обработки событий и агрегации данных.
*   **`parse_replay(replay_path, output_path)`**: Главный пайплайн. Загружает реплей, запускает парсинг событий, финализирует данные, нормализует координаты и сохраняет JSON.
*   **`_parse_events()`**: Сортирует все `tracker_events` и `game_events` по времени и маршрутизирует их по хендлерам.
*   **Хендлеры событий**:
    *   `_handle_init_event`: Обрабатывает `UnitInitEvent` / `UnitBornEvent` (разделяет здания, нейтралов и юнитов).
    *   `_handle_type_change`: Ловит `UnitTypeChangeEvent` (трансформации зданий).
    *   `_handle_positions_event`: Обрабатывает `UnitPositionsEvent` (обновление координат).
    *   `_handle_death_event`: Обрабатывает `UnitDiedEvent` (смерть/уничтожение).
*   **`_normalize_all_positions()`**: Сдвигает все координаты так, чтобы центр карты стал `(0, 0)`.
*   **`_build_result()`**: Формирует итоговый словарь с `metadata`, `buildings` и `units`.

### 3. Точки входа
*   **`parse_replay_unified()`**: Обертка для вызова парсера из других модулей.
*   **`__main__`**: Консольный запуск (CLI) с ручным вводом пути к реплею.

## ️ Важные технические детали (Для разработчиков)

Если вы планируете изменять или использовать данные из этого модуля, обратите внимание на следующие нюансы:

1.  **Фильтрация микро-движений (Оптимизация данных)**
    В методе `_handle_positions_event` реализован фильтр: если юнит сместился менее чем на `0.05` единиц за `0.5` секунды, новая точка **не сохраняется**. 
    *Важно:* На выходе мы получаем не "сырые" покадровые данные, а сглаженные траектории. Это сильно экономит размер JSON, но может скрыть микро-контроль (micro) на месте.
2.  **Потеря абсолютных координат карты**
    Метод `_normalize_all_positions` центрирует карту по `min/max` записанных точек, а не по реальным размерам карты SC2. Оригинальные координаты Blizzard не сохраняются. Все координаты на выходе — относительные, где `(0,0)` — это геометрический центр всех зафиксированных событий.
3.  **Жесткая логика Z-координаты (Высоты)**
    Метод `_get_z_coordinate` не читает рельеф карты. Он возвращает `7.0` для летающих юнитов (проверяет суффикс `Flying` или ищет по жестко заданному списку `flying_types`) и `0.0` для наземных.
4.  **Обработка нейтральных зданий**
    В `_handle_init_event` есть специфичная логика для `UnitBornEvent` без контроллера (игрока) — такие объекты помечаются как нейтральные здания (например, Xel'Naga Towers, Minerals, Vespene).
5.  **Внешние зависимости**
    *   `sc2reader`: Основная библиотека для чтения `.SC2Replay`.
    *   `ui_processing`: Кастомный модуль проекта (функция `ui_progress`). Если файл используется вне основного проекта, этот импорт нужно закомментировать или заменить на заглушку, иначе упадет `ImportError`.
6.  **Опечатка в имени файла**
    Файл называется `parce_replay.py` (вместо `parse`). Опечатка историческая, **не рекомендуется** переименовывать файл без рефакторинга всех импортов в проекте (`from parce_replay import ...`).
