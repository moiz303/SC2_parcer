# Документация: `controller_backend.py`

## Назначение
Центральный оркестратор (связующее звено) между фронтендом и тяжелым бэкендом. Управляет жизненным циклом задачи рендеринга (Job), запускает аналитические скрипты, формирует данные для Blender, инициирует headless-рендер и в реальном времени собирает прогресс со всех этапов для передачи в UI.

## Описание
Модуль выступает в роли «дирижера». Он принимает состояние от фронтенда, последовательно вызывает парсер реплея и алгоритм поиска главной битвы, а затем запускает Blender как отдельный субпроцесс. Ключевая особенность — **перехват стандартного вывода (stdout)**. Модуль перехватывает как локальные `print()` из Python-скриптов, так и вывод из subprocess Blender, парсит специальные маркеры (`__UI__`, `__GLB__`, `Video append frame`) и конвертирует их в колбэки прогресса.

## ️ Структура (High-Level Overview)

### 1. Модели данных (Dataclasses)
*   **`JobConfig`**: Конфигурация задачи. Хранит пути к реплею, ассетам (юниты, здания, текстуры), настройки длительности и флаги рендера.
*   **`JobResult`**: Итоговый статус задачи. Содержит флаг `success`, путь к итоговому видео, сообщение об ошибке и метаданные (включая результаты анализа и пути к временным JSON).

### 2. Главный оркестратор: `BackendController`
*   **`build_job_config()`**: Маппинг словаря от фронтенда в строгий объект `JobConfig`.
*   **`run_job()`**: Главный пайплайн. Запускает локальный парсинг, ищет главную битву, готовит payload для Blender и стартует рендер.
*   **`prepare_render_payload()`**: Формирует итоговый JSON-словарь, который будет передан в Blender-скрипт.
*   **`resolve_output_video_path()`**: Умное разрешение пути для сохранения видео (поддерживает папки, файлы с разными расширениями, создает дефолтный `result.mp4`).

### 3. Перехватчики и трекеры (I/O & Progress)
*   **`_StdoutUiTap`**: Кастомный враппер для `sys.stdout`. Перехватывает локальные `print()` в текущем процессе, ищет строки с префиксом `__UI__` и отправляет их в колбэк прогресса.
*   **`_RenderProgressTracker`**: Парсер вывода Blender. Ищет строки вида `Video append frame N`, чтобы вычислять «честный» процент завершения рендера.

### 4. Утилиты запуска subprocess
*   **`find_blender_executable()`**: Ищет исполняемый файл Blender (в ENV, в жестко прописанных путях Windows или в системном `PATH`).
*   **`run_blender_render()`**: Запускает Blender через `subprocess.Popen`, читает stdout построчно, маршрутизирует строки по трекерам и обрабатывает таймауты.

## Важные технические детали (Для разработчиков)

### 1. Протокол общения через Stdout (`__UI__` и `__GLB__`)
Вся коммуникация о прогрессе построена на перехвате текста. 
*   Локальные скрипты (или сам Blender) должны писать в stdout строки вида: `__UI__ {"stage": "import_scene", "progress": 45, "message": "Загрузка..."}`.
*   Для сигналов о готовности 3D-моделей используется префикс `__GLB__`.
*   *Важно:* Если вы добавляете новые этапы, они должны соответствовать этому JSON-формату, иначе `_handle_ui_line` их отбросит.

### 2. Двойной перехват stdout
В `run_job` используется `contextlib.redirect_stdout(stdout_tap)`. Это означает, что когда `parce_replay` и `find_main_battle` вызывают свои внутренние `ui_progress` (которые делают `print`), вывод не падает в консоль, а перехватывается `_StdoutUiTap` и мгновенно уходит во фронтенд. 
*Нюанс:* Это работает только для синхронного кода внутри основного процесса Python. Для Blender используется отдельный парсинг `subprocess.PIPE`.

### 3. «Честный» прогресс рендера
Blender не отдает прогресс в процентах напрямую. `_RenderProgressTracker` вычисляет его математически, считая строки `Video append frame N`. Если в `render_payload` не передано `duration_frames`, трекер выведет предупреждение и процент рендера для фронтенда будет `0`.

### 4. Поиск исполняемого файла Blender
Функция `find_blender_executable` содержит жестко зашитые пути для Windows (`Blender 4.4`, `Blender 5.0`). Если вы переносите проект на другую ОС или версию Blender, **обязательно** обновите этот список или передавайте путь через переменную окружения `BLENDER_EXE`.

### 5. Таймауты и обработка зависаний
В `run_blender_render` установлен жесткий таймаут **900 секунд** (15 минут). Если Blender не успевает, процесс принудительно убивается (`process.kill()`), и возвращается ошибка. Это спасает от зависших фоновых процессов, если рендер сложной сцены пошел не по плану.

### 6. Временные файлы
Все промежуточные JSON (`unified_...json`, `render_payload_...json`) сохраняются в папку `temp` внутри `base_dir`. Модуль не очищает их автоматически после завершения, что может потребовать настройки периодической очистки (cron/скрипт), если приложение используется интенсивно.
