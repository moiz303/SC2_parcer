# Архитектура Desktop-приложения генерации анимаций из реплеев

## 1. Общее описание и цели проекта
Проект представляет собой настольное Windows-приложение, которое принимает файл игрового реплея (`.SC2Replay`), генерирует на его основе трёхмерную сцену и анимацию в Blender, а затем сохраняет результат в виде видеофайла.  
Приложение ориентировано на пользователей, работающих с реплеями StarCraft II, но в будущем будет расширено на другие форматы.

**Ключевые сценарии использования:**
- Пользователь выбирает реплей, указывает параметры анимации и расположение ассетов, запускает генерацию и получает готовое видео.
- Пользователь может просмотреть краткое превью результата после завершения.

---

## 2. Архитектурный обзор
Приложение построено по модульному принципу, где весь «умный» код написан на Python, а Blender используется как внешний движок рендеринга, запускаемый в фоновом режиме.  
Высокоуровневая диаграмма компонентов:

[GUI (Tkinter)] <-- вызовы --> [Controller Backend] <-- subprocess --> [Blender Executor]  
│  
├─ ReplayParser  
├─ ConfigManager  
└─ JobManager

- **GUI** – отвечает только за отображение и сбор пользовательского ввода. Не содержит бизнес-логики.
- **Controller Backend** – управляет жизненным циклом задачи: валидация, парсинг реплея, запуск Blender, передача прогресса.
- **Blender Executor** – абстракция, скрывающая способ запуска Blender (сейчас локальный, в будущем удалённый).

Приложение является монолитным (все компоненты живут в одном процессе), но логически разделено для удобства поддержки и тестирования.  
Взаимодействие между GUI и контроллером – прямые вызовы методов и сигналы (observer-подобная модель на базе `tkinter` `StringVar`/`after` или кастомных событий).  
Blender всегда запускается как отдельный процесс ОС через `subprocess`.

**Обоснование выбора:**
- Минимум движущихся частей, лёгкая отладка.
- Нет сетевых задержек – всё выполняется локально.
- Пользователь получает мгновенную обратную связь.

## 3. Детальное описание компонентов

### 3.1 Frontend (GUI на Tkinter)
**Технология:** `tkinter` + `ttk` (стандартная библиотека Python).  
**Основные окна (фреймы):**
1. **Приветствие / начальный экран** – информация о программе, версия Blender-зависимости, кнопка «Начать».
2. **Настройка задачи** – выбор файла реплея (через стандартный диалог или ручной ввод пути), выбор длительности анимации (выпадающий список: 30/45/60/75 сек, по умолчанию 45), выбор места сохранения итогового видео.
3. **Выбор ресурсов** – указание директорий с моделями юнитов и зданий, а также с текстурами (два поля с кнопками обзора).
4. **Мониторинг прогресса** – индикатор выполнения (`Progressbar`), лог сообщений от Blender, кнопка «Отмена» (если поддерживается).
5. **Результат** – сообщение об успехе, путь к видео, область предпросмотра (например, встроенный видеоплеер или открытие системным проигрывателем), кнопка «Новый реплей».

**Передача команд:** GUI собирает все настройки в объект `JobConfig` и передаёт его в `JobManager.start(job_config)`.  
Обратная связь от контроллера: через механизм колбэков или обновление привязанных `StringVar` (для лога, прогресса, статуса). Контроллер вызывает методы GUI, запланированные через `after`, чтобы избежать блокировки главного потока.

### 3.2 Controller Backend
Набор Python-модулей без графической зависимости. Ядро – `JobManager`.

- **`ConfigManager`** – хранит пути к Blender, пути по умолчанию, настройки приложения, загружает/сохраняет конфигурацию (ini/json).
- **`ReplayParser`** – преобразует исходный `.SC2Replay`-файл в JSON-структуру, пригодную для передачи в Blender. В будущем – расширяемый интерфейс для других форматов.
- **`JobManager`** – управляет очередью задач (пока одной). Получает `JobConfig`, валидирует входные данные, проверяет наличие Blender (версия ≥ 5.0.1), запускает задачу в отдельном потоке (`threading.Thread`), контролирует выполнение, транслирует прогресс.

**Проверка Blender:**  
Метод `ConfigManager.find_blender()` ищет исполняемый файл `blender.exe` в:
- Явно заданном пользователем пути (сохранённом в конфиге).
- Стандартных путях установки (`Program Files`, `Blender Foundation`).
- Переменной `PATH`.  
Если версия ниже требуемой или файл не найден – GUI отображает ошибку и ссылку на официальный сайт.

### 3.3 Blender Executor
Интерфейс `BlenderExecutor` определяет метод `execute(blend_script_path: Path, json_data_path: Path, output_dir: Path, duration: int) -> ExecutionResult`.

**Локальная реализация (`LocalBlenderExecutor`):**
- Формирует аргументы командной строки:  
  `blender --background --python <script.py> -- <json_file> <output_dir> <duration>`
- Запускает процесс через `subprocess.Popen` с захватом `stdout` и `stderr`.
- В отдельном потоке читает `stdout` построчно, парсит маркеры: `PROGRESS <число>`, `DONE <путь_к_видео>`, `ERROR <текст>`.
- Возвращает `ExecutionResult` с путём к итоговому видео или ошибкой.
- При возникновении ошибки внутри Blender (ненулевой код возврата) парсит `stderr` и формирует понятное сообщение.

**Будущая удалённая реализация (`RemoteBlenderExecutor`):**
- Отправляет JSON-задание и файл реплея на удалённый сервер через REST API.
- Получает прогресс через WebSocket или Server-Sent Events.
- Загружает готовое видео по ссылке.
- Абстракция позволяет переключиться без изменения GUI и контроллера.

### 3.4 Вспомогательные модули
- **`Logger`** – обёртка над стандартным `logging` для записи событий в файл и дублирования в GUI-лог.
- **`Updater`** (опционально) – проверка обновлений на GitHub.

## 4. Потоки данных и интерфейсы ввода-вывода (I/O)

### 4.1 Входные данные
| Элемент           | Формат            | Источник                       |
|-------------------|-------------------|--------------------------------|
| Файл реплея       | `.SC2Replay`      | Файловая система, пользователь |
| Длительность      | число (30-75)     | Выбор пользователя             |
| Папка моделей     | путь к директории | Файловая система               |
| Папка текстур     | путь к директории | Файловая система               |
| Выходная папка    | путь к директории | Пользователь                   |

**Процесс парсинга:**
1. `ReplayParser.parse(replay_path)` читает реплей, извлекает необходимые данные (события, команды, состояние игры) и формирует JSON-файл (например, `job_<timestamp>.json`).
2. JSON сохраняется во временную директорию приложения.

### 4.2 Выходные данные
- Основной результат: видеофайл в формате MP4 (кодек H.264) или другом заданном, сохраняется в указанную пользователем папку.
- Дополнительно: лог-файл сессии в папке `logs/`.

### 4.3 Взаимодействие с Blender-скриптом
- **Передача данных:** через временный JSON-файл. Скрипт при запуске получает путь к JSON через аргумент командной строки (`sys.argv[6]` после `--`).
- **Настройки:** длительность, пути к ассетам, выходная директория – также через аргументы или JSON.
- **Протокол stdout:** скрипт обязан выводить маркированные строки:
  - `PROGRESS <целое 0-100>` – процент выполнения.
  - `DONE <полный_путь_к_видео>` – успешное завершение.
  - `ERROR <сообщение>` – ошибка, после которой процесс может завершиться.
- Всё остальное, что выводит скрипт, считается отладочной информацией и может записываться в лог.

### 4.4 Внутреннее взаимодействие GUI ↔ Controller
Синхронные вызовы методов (запуск задачи – неблокирующий, выполняется в фоновом потоке) и колбэки для обновления UI. Колбэки выполняются в главном потоке через `tkinter.after` для потокобезопасности.

## 5. Описание работы Blender-скрипта

**Расположение:** `scripts/generate_animation.py` (в ресурсах приложения, распаковывается во временную папку при запуске).

**Назначение:** Получить JSON с данными реплея, создать сцену в Blender, настроить камеры, освещение, модели юнитов/зданий из указанных папок, анимировать в соответствии с событиями реплея, отрендерить анимацию и сохранить в видеофайл.

**Входные параметры (через sys.argv после `--`):**
1. `<путь к JSON>`
2. `<выходная директория>`
3. `<длительность анимации в секундах>`
4. `<путь к папке моделей>`
5. `<путь к папке текстур>`

**Ожидаемое поведение:**
- Установить сцену, загрузить ассеты.
- Настроить параметры рендера согласно длительности (FPS = 30, количество кадров = длительность * 30).
- Проигрывать анимацию.
- Выводить `PROGRESS <процент>` по мере готовности кадров (например, каждые 10%).
- По завершении рендера записать видео по указанному пути.
- Вывести `DONE <путь к видео>`.

**Требования к окружению:**
- Blender 5.0.1 или новее.
- Права на чтение файлов моделей/текстур и запись в выходную директорию.

## 6. Интеграция интерфейса пользователя

### 6.1 Экраны и навигация
- **Приветствие** → кнопка «Начать».
- **Настройка задачи** → проверка заполнения всех полей, кнопка «Далее».
- **Выбор ресурсов** → проверка заполнения всех полей, кнопка «Запустить».
- **Прогресс** – модальное или встроенное окно с прогресс-баром и логом
- **Результат** → предпросмотр результата, кнопка «Новый реплей» (возврат к экрану 1) и кнопка «Готово».

### 6.2 Обработка длительных операций
- Генерация JSON и запуск Blender выполняются в фоновом потоке.
- GUI не блокируется; для предотвращения повторного запуска кнопка «Запустить» деактивируется до завершения или ошибки.
- Прогресс обновляется через вызов `root.after(0, callback)`.

### 6.3 Валидация
- Наличие файла реплея и его расширение `.SC2Replay`.
- Существование выбранных папок с ассетами.
- Проверка наличия Blender и его версии (при старте приложения и перед запуском).
- Проверка свободного места на диске (опционально, пока не реализовано).

## 7. Обработка ошибок и логирование

### 7.1 Ошибки этапа подготовки
- Не найден Blender/неподходящая версия → сообщение с инструкцией в GUI, задача не запускается.
- Неверный формат реплея → нет возможности выбрать файл (интерфейс выбора принимает только .SC2Replay).

### 7.2 Ошибки выполнения Blender
- Ненулевой код возврата → парсим `stderr`, отображаем последние строки ошибки в логе GUI.
- Тайм-аут выполнения (если задача висит дольше разумного) → принудительное завершение процесса с уведомлением.
- Blender скрипт вывел `ERROR ...` → отображаем сообщение и останавливаем задачу.

### 7.3 Логирование
- Файловый лог (`logs/app.log`) с ротацией при достижении 5 МБ.
- Уровни: `DEBUG` в файл, `INFO` и выше – в GUI-консоль.
- Лог содержит: вызовы методов, аргументы, stdout/stderr Blender, исключения.

## 8. Развёртывание и зависимости

**Целевая ОС:** Windows 10/11 (x64).  
**Зависимости:**
- Python 3.11+ (если не упаковано в exe).
- Blender 5.0.1+ (требуется отдельная установка, не входит в дистрибутив).
- Стандартные библиотеки Python: `tkinter`, `json`, `subprocess`, `threading`, `pathlib`, `logging`, `sys`, `os`.

**Упаковка:** PyInstaller создаёт один `.exe` файл, который включает интерпретатор Python, все модули и ресурсы (скрипт для Blender, иконки).  
Ресурсы при запуске распаковываются во временную папку (`%TEMP%/ReplayAnimator/`).

**Установка:**  
- Пользователь скачивает `.exe` с GitHub Releases.
- При первом запуске приложение просит указать путь к `blender.exe` (если не найден автоматически) и запоминает его в `config.ini` рядом с exe.
- Если Blender не установлен – выводится ссылка на официальный сайт.

**Portable-режим (пока не реализован):** Возможность поместить `blender.exe` в подпапку `blender/` рядом с приложением – тогда программа будет использовать его без дополнительных настроек.

## 9. Направления развития

### 9.1 Клиент-серверная архитектура
- Замена `LocalBlenderExecutor` на `RemoteBlenderExecutor` без изменения основной логики.
- Сервер принимает задания через REST (загрузка реплея и конфигурации), имеет очередь на базе Redis + Celery, запускает Blender в Docker-контейнере с GPU.
- Клиент получает прогресс через WebSocket, готовое видео скачивается по HTTP.

### 9.2 Расширение поддерживаемых форматов реплеев
- Абстракция `BaseReplayParser`, фабрика по расширению файла.
- Добавление парсеров для других игр (например, `.RA3Replay` для Command&Conquer: Red Alert 3).

### 9.3 Улучшенный предпросмотр
- Встроенный видеоплеер на базе `tkinter` + `opencv-python` или открытие системного плеера.

### 9.4 Очередь локальных задач
- Возможность добавлять несколько реплеев, обрабатывать их последовательно.

---

## 10. Заключение
Данная архитектура обеспечивает быструю разработку, простоту сопровождения и естественный путь эволюции от локального инструмента к облачному сервису. Жёсткая зависимость от Blender оправдана предметной областью и успешно преодолевается понятными инструкциями для пользователя и гибким механизмом поиска исполняемого файла.
