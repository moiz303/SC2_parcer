# Replay Master: Архитектура и Глобальная Документация

## О проекте
**Replay Master** — это комплексное десктопное приложение для автоматического парсинга реплеев StarCraft II, алгоритмического поиска главного сражения и генерации кинематографичной 3D-анимации итоговой битвы. 

Проект связывает в себе парсинг бинарных данных RTS, математическую аналитику, графический интерфейс (GUI) и headless-рендеринг в Blender.

---

## Глобальный Пайплайн (Data Flow)
Вся работа приложения строится вокруг последовательного преобразования данных. От клика пользователя до готового `.mp4` файла данные проходят 4 строгие стадии:

1. **Сбор и Парсинг (`parce_replay.py`)**: Сырой `.SC2Replay` превращается в унифицированный `unified.json`.
2. **Аналитика (`find_main_battle.py`)**: Алгоритм находит временной интервал и геометрический центр самого массового боя.
3. **Оркестрация (`controller_backend.py`)**: Бэкенд формирует `render_payload.json` и запускает Blender как субпроцесс.
4. **3D-Движок (`import_to_blender.py`)**: Blender импортирует ассеты, анимирует тысячи юнитов, настраивает камеру и рендерит видео.

## Модульная Архитектура

Проект разделен на 4 логических слоя:

### 1. Frontend Layer (UI)
*   **`frontend_main.py`** *(лежит в `frontend/`)*
    *   **Роль:** Точка входа для пользователя. Мастер (Wizard) на `tkinter`.
    *   **Особенности:** Кастомные виджеты (`RoundedButton`, `SmoothProgressBar`), фоновый предпросмотр видео через `OpenCV`, автопоиск папок с модами.
    *   **Связь:** Запускает `BackendController` в `daemon`-потоке, получая обновления через колбэки.

### 2. Orchestration Layer (Backend)
*   **`controller_backend.py`**
    *   **Роль:** "Дирижер". Управляет жизненным циклом задачи (Job).
    *   **Особенности:** Перехват `sys.stdout` (`_StdoutUiTap`) для получения прогресса от Python-скриптов и парсинг вывода `subprocess` Blender (`_RenderProgressTracker`).
    *   **Связь:** Вызывает аналитические модули напрямую, а рендер-скрипт запускает через CLI.

### 3. Analytics Layer (Data & Math)
*   **`parce_replay.py`**
    *   **Роль:** Низкоуровневый парсер `sc2reader`. Превращает события в Dataclasses (`Unit`, `Building`), фильтрует микро-шум движений, центрирует координаты карты относительно `(0,0)`.
*   **`find_main_battle.py`**
    *   **Роль:** Алгоритм поиска эпицентра. Использует метод скользящего окна (30 сек) и пространственную фильтрацию (Grid-based noise reduction) для поиска пика смертей.

### 4. 🎬 3D Engine Layer (Blender)
*   **`import_to_blender.py`** *(Сердце проекта)*
    *   **Роль:** Переводит JSON в 3D-сцену. Математика траекторий, NLA-анимации, процедурные материалы, кинематографическая камера (облет вокруг центра битвы).
    *   **Особенности:** Поддержка двух режимов (CLI для бэкенда и Mock-данные для ручной отладки внутри Blender).
*   **`parce_m3_models.py`** *(Standalone утилита)*
    *   **Роль:** Пакетный конвертер оригинальных `.m3` моделей в `.blend` для подготовки ассетов. Не участвует в основном пайплайне.

## Контракты Данных (JSON Schema)

Вся коммуникация между модулями (кроме UI) идет через JSON-файлы во временной папке `temp/`.

### 1. `unified.json` (Выход парсера -> Вход аналитики и Blender)
Жесткий контракт. Изменение структуры сломает downstream.
```json
{
  "metadata": { "duration": "...", "map": "...", "players": [...] },
  "buildings": [
    { "entity_id": 1, "type": "CommandCenter", "owner_race": "Terran", 
      "init_frame": 100, "died_frame": null, 
      "frames": [{"frame": 100, "x": 0.0, "y": 0.0, "z": 0.0}], 
      "transforms": [{"frame": 5000, "from_type": "...", "to_type": "..."}] }
  ],
  "units": [
    { "entity_id": 2, "type": "Marine", "is_army": true, 
      "init_frame": 200, "died_frame": 4500, 
      "positions": [{"frame": 200, "x": 10.0, "y": -5.0, "z": 0.0}] }
  ]
}
```
 **Важно:** Координаты `x` и `y` **относительные**. `(0,0)` — это геометрический центр всех зафиксированных событий на карте, а не угол карты Blizzard.

### 2. `render_payload.json` (Выход бэкенда -> Вход Blender)
Содержит пути к ассетам и результаты аналитики.
```json
{
  "unified_json_path": "C:/.../temp/unified_xxx.json",
  "units_path": "...", "buildings_path": "...", "textures_path": "...",
  "analysis": {
    "success": true, "start_frame": 14513, "end_frame": 15013,
    "duration_frames": 500, "center_x": 36.0, "center_y": 5.0
  },
  "output_path": "C:/.../result.mp4"
}
```

## Протокол межпроцессного взаимодействия (IPC)

Поскольку Blender запускается как отдельный процесс (`subprocess.Popen`), обмен прогрессом происходит через **стандартный вывод (stdout)**.

1. **`__UI__` Protocol**: Любой скрипт (Python или Blender) может отправить прогресс во фронтенд, написав в stdout:
   `__UI__ {"stage": "import_scene", "progress": 45, "message": "Загрузка..."}`
   Бэкенд перехватывает это и передает в UI.
2. **`__GLB__` Signal**: Сигнал о том, что 3D-сцена готова и можно показывать превью (используется экспорт в `.obj` / `.glb`).
3. **Render Frames**: Бэкенд парсит нативный лог Blender `Video append frame N`, чтобы вычислять "честный" процент рендера для UI.

## ️ Технические Требования и Зависимости

### Критические требования:
*   **Blender 4.4+ (СТРОГО)**: Скрипт `import_to_blender.py` использует новый API *Layered Actions* (`action.layers`). На версиях ниже 4.4 анимации будут обрабатываться некорректно или скрипт упадет.
*   **Python 3.10+**: Используются современные фичи (Dataclasses, Type Hinting, `match/case` паттерны в логике).

### Основные библиотеки:
*   `sc2reader` — Парсинг `.SC2Replay`.
*   `tkinter` / `ttk` — GUI.
*   `opencv-python` (`cv2`) + `Pillow` — Превью видео в UI.
*   `bpy` (Blender Python API) — 3D-движок.

## Режимы Запуска

### 1. Продакшн (Для пользователя)
Запуск через собранный `.exe` (или корневой `.py` файл-обертку) -> Открывается GUI -> Пользователь выбирает файлы -> Генерация видео.

### 2. Ручная отладка 3D-сцены (Для разработчика)
Если нужно быстро проверить, как Blender читает конкретный JSON, без запуска UI и бэкенда:
1. Открыть Blender.
2. Перейти во вкладку *Scripting*.
3. Открыть `import_to_blender.py`.
4. Заполнить моковые пути в **Разделе 1** (в начале файла).
5. Нажать *Run Script*.

### 3. Подготовка ассетов (Standalone)
Запуск `parce_m3_models.py` **внутри среды Blender** (через Text Editor или CLI `blender -b -p parce_m3_models.py`) для конвертации папки `.m3` в `.blend`.

## Исторические и Архитектурные Нюансы (Для разработчиков)

1. **Опечатка `parce_`**: Файлы `parce_replay.py` и `parce_m3_models.py` содержат опечатку (вместо `parse_`). Она историческая, **не рекомендуется** переименовывать файлы без рефакторинга всех импортов и путей в CI/CD.
2. **Очистка `temp/`**: Фронтенд при старте и закрытии безжалостно удаляет папку `temp` в корне проекта. Не храните там пользовательские данные.
3. **Математика поворота (Shortest Path)**: В `import_to_blender.py` реализована нормализация углов (`math.pi`), чтобы юниты не крутились на 350 градусов, когда им нужно повернуться на 10.
4. **Жадный алгоритм стройки**: В `preprocess_data` скрипт сам находит рабочих (SCV/Drone), которые подошли к зданию, "обрезает" их траекторию и искусственно "убивает" Дронов в момент стройки (так как они сливаются с зданием в SC2).
5. **Отсутствующая `DEST_DIR`**: В `parce_m3_models.py` в настройках нужно вручную прописать `DEST_DIR`, иначе скрипт упадет с `NameError`.
